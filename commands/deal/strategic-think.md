## Thinking rules (always apply)

Before starting, anchor to these principles:

- **Find the root cause and solve for that** — not the symptom (${user_config.company} mantra)
- **Win-win or no deal** — any solution must satisfy both sides' real needs (${user_config.company} mantra)
- **Extreme Ownership** — own the outcome completely, even what isn't directly your fault. No victim mindset, no "because of them."
- **Slower today, faster forever** — think it through properly now; rushing creates rework (${user_config.company} mantra)
- **Prioritize and execute** — one constraint at a time. Never optimise everything at once.

## Step 2 — Current Reality: Find the Root Cause (CRT Logic)

Connect UDEs using if→then causality. Work backwards from each UDE until you find the core driver — the one cause most UDEs trace back to.

For each causal link, validate three tests:
- **Causality test:** Does this cause reasonably produce that effect?
- **Assumption test:** What hidden belief makes this link hold?
- **Entity test:** Are both statements clear and specific enough to stand alone?

**Causal chain (if→then):**

```
UDE 1 ←── Intermediate Effect ←── Intermediate Effect ←── Root Cause
UDE 2 ←────────────────────────────────────────────────── Root Cause
UDE 3 ←── Intermediate Effect ←── Root Cause
```

## Step 3 — Reveal the Hidden Conflict (Evaporation Cloud)

If there is tension — two reasonable actions that cannot coexist — map the cloud.

A cloud exists when: both actions make sense, both serve real needs, and you cannot do both at the same time.

| Element | Statement |
|---------|-----------|
| **Common goal** | What both sides ultimately want |
| **Need A** (to achieve the goal) | What one side needs |
| **Need B** (to achieve the goal) | What the other side needs |
| **Action A** (want) | What one side is doing or pushing for |
| **Action B** (want) | What the other side is doing or pushing for |
| **The conflict** | Why A and B cannot coexist right now |

**Win-win test:** Are both needs genuinely required, or is one side defending a policy/assumption rather than a real need?

If the conflict oscillates (keeps coming back), map the **Druid loop**: what behaviour does the goal violation cause, which reinforces the conflict, which produces more rework, which restores the violation?
