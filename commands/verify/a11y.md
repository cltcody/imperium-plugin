# Accessibility Review

Audit the user-facing surface for WCAG mechanics. Default scope is **diff** — the uncommitted changes and branch delta; pass `full` for the whole repo. This command is **stack-agnostic**: which components have a UI, and any dev-server command for the runtime pass, come from the project's `STACK.md`, not from this file. It is also **recurring by design**: findings are reconciled against the findings ledger (command id `verify:a11y`) so re-runs stay quiet about what you already know, and `--delta` reports only what changed. Scope split: this is the **WCAG-mechanics pass** — flow-level UX (friction, missing states, copy) belongs to the `ux-reviewer` agent inside `/cc:verify:design`; no overlap.

## Steps

1. **Resolve scope (and the stack).** Read `STACK.md` and resolve per `${CLAUDE_PLUGIN_ROOT}/references/dev/stack-resolution.md`. Identify which components have a **user-facing surface** and detect each one's UI idiom: HTML templates (Django/Jinja/Rails/Astro), JSX/TSX (React/Next.js), Vue or Svelte SFCs, React Native (Expo). If **no component has a user-facing surface** (API-only service, library, headless CLI), report "n/a — no user-facing surface" and stop. No `STACK.md` → auto-detect once and recommend `/cc:setup:stack`.
   - `diff` (default): collect `git diff` (unstaged + staged) plus `git diff main...HEAD`; keep UI-relevant files (templates, components, styles, theme/token files). Empty → report "nothing to scan" and stop.
   - `full`: scan every UI component's source directories.

2. **Static rule catalog.** Run each applicable rule over the in-scope files — the rule id is the ledger `rule`. Skip rules that don't apply to the detected idiom (e.g. `a11y.doc-lang` on React Native) and mark them n/a in the rule strip. The greps are illustrative starting points — adapt the attribute idiom to the framework (`alt=` in HTML/JSX, `:alt` bindings in Vue, RN props), and remember multi-line attributes make single-line greps approximate: **confirm every hit at its file:line before reporting** (e.g. re-grep with `-A 3` and read the element). Report honest "no finding" per rule scanned — never invent issues.

   **`a11y.img-alt` — images and icons without a text alternative.** Content-bearing `<img>`/`<Image>`/inline `<svg>` needs `alt` or an ARIA label; decorative ones need explicit `alt=""` or `aria-hidden="true"`. Example greps:
   ```bash
   grep -rn "<img\|<Image" <src-dirs> --include="*.html" --include="*.jsx" --include="*.tsx" --include="*.vue" --include="*.svelte" | grep -v "alt=\|:alt\|aria-hidden"
   grep -rn "<svg" <src-dirs> | grep -v "aria-hidden\|aria-label\|role="
   ```
   True positive: an image carrying content or function with no text alternative at all. Clearly decorative and merely missing the explicit empty `alt` → LOW.

   **`a11y.form-label` — form inputs without an associated label.** Every visible `<input>`/`<select>`/`<textarea>` needs a programmatic label: wrapping `<label>`, `id` paired with `for=`/`htmlFor=`, or `aria-label(ledby)`. Placeholder text is not a label. Example grep:
   ```bash
   grep -rn "<input\|<select\|<textarea" <src-dirs> | grep -v "type=[\"']hidden\|aria-label"
   ```
   For surviving hits, verify the association (wrapping label or a matching `for=`/`htmlFor=` elsewhere in the file). True positive: a control a screen reader announces as just "edit text".

   **`a11y.interactive-role` — click handlers on non-interactive elements.** A `div`/`span` with a click handler needs all three of `role`, `tabindex="0"`, and a key handler (Enter/Space) — a real `<button>`/`<a href>` is always the better fix. Example grep:
   ```bash
   grep -rn "<div[^>]*onClick\|<span[^>]*onClick\|<div[^>]*@click\|<span[^>]*@click\|<div[^>]*on:click\|<span[^>]*on:click" <src-dirs>
   ```
   True positive: a hit missing any of the three — keyboard-only users cannot activate it (WCAG 2.1.1).

   **`a11y.doc-lang` — missing document language** (web only). The document shell that renders `<html>` must carry `lang` (WCAG 3.1.1 — wrong screen-reader voice otherwise). Example grep:
   ```bash
   grep -rln "<html" <src-dirs> --include="*.html" --include="*.jsx" --include="*.tsx" --include="*.astro" | xargs grep -L "lang=" 2>/dev/null
   ```
   True positive: the app shell/layout without `lang` — one finding per shell, not per page that includes it.

   **`a11y.heading-order` — heading-level jumps.** Example grep:
   ```bash
   grep -rn "<h[1-6]" <src-dirs> --include="*.html" --include="*.jsx" --include="*.tsx" --include="*.vue" --include="*.svelte"
   ```
   Judge at the **composed page** level, not per component fragment — a card partial that starts at `h3` is fine if the layout supplies `h1`/`h2`. True positive: a real jump (h1 → h3 with no h2) on a rendered page (WCAG 1.3.1).

   **`a11y.tabindex` — positive tabindex.** Any value ≥ 1 hijacks focus order (WCAG 2.4.3); `0` and `-1` are fine. Example grep:
   ```bash
   grep -rn "tabindex=[\"'][1-9]\|tabIndex={[1-9]" <src-dirs>
   ```
   Every confirmed hit is a true positive.

   **`a11y.focus-modal` — missing focus management in modals/dialogs.** An open dialog needs all three: focus trapped inside, Escape closes it, focus returns to the trigger on close. Example locator:
   ```bash
   grep -rlni "modal\|dialog\|drawer\|popover" <src-dirs> --include="*.tsx" --include="*.jsx" --include="*.vue" --include="*.svelte"
   ```
   Read each hit. Native `<dialog>` + `showModal()`, or a maintained primitive (Radix, Headless UI, focus-trap), satisfies all three; a hand-rolled positioned overlay almost never does. True positive: Tab escapes into the page behind the overlay, or Escape does nothing.

   **`a11y.media-autoplay` — autoplaying media without controls.** Example grep:
   ```bash
   grep -rn "autoplay\|autoPlay" <src-dirs> | grep -v "controls"
   ```
   True positive: audible media that autoplays with no visible way to pause/stop (WCAG 1.4.2). Muted looping background video is MEDIUM at most — still needs a pause affordance and `prefers-reduced-motion` respect (2.2.2).

   **`a11y.contrast` — color contrast.** Where theme/token files make foreground/background pairs derivable, compute the WCAG ratio; otherwise never guess. Example grep:
   ```bash
   grep -rn "#[0-9a-fA-F]\{3,8\}" <src-dirs> --include="*token*" --include="*theme*" --include="*.css" --include="*.scss" | grep -i "color\|background\|text\|surface\|bg\|fg"
   ```
   AA thresholds: 4.5:1 normal text, 3:1 large text and UI components. True positive: a derived pair below threshold, reported with both hex values and the computed ratio. Pairs not derivable statically → flag the key pairs (body text on page background, primary button label) for the runtime pass or a manual check — as ℹ️ INFO, not a rated finding.

   **`a11y.rn-props` — React Native touchables without accessibility props** (RN only). Example greps:
   ```bash
   grep -rn "<Touchable\|<Pressable" <src-dirs> --include="*.tsx" --include="*.jsx" | grep -v "accessibilityLabel\|accessibilityRole"
   grep -rn "accessible={false}" <src-dirs> --include="*.tsx" --include="*.jsx"
   ```
   True positive: an icon-only touchable with neither `accessibilityLabel` nor a text child (announced as nothing). A text-child button missing only `accessibilityRole="button"` → MEDIUM.

3. **Optional runtime pass.** Run **only if** `STACK.md` resolves a `dev` command for a UI component **and** axe tooling is already installed (never install tooling unprompted). Check, then run against key routes — the home/entry route plus routes the diff touches — and fold results into the findings list with rule ids prefixed `axe.<rule-id>` (same ledger treatment):
   ```bash
   cd <ui component working_dir> && npm ls @axe-core/playwright 2>/dev/null || npx --no-install lighthouse --version 2>/dev/null || echo "no axe/Lighthouse tooling installed"
   ```
   Neither available, or no `dev` command → **skip with an explicit note** in the report ("runtime pass: SKIPPED — <reason>"); never imply runtime coverage that didn't happen.

4. **Rate every finding** per the canonical ladder in `${CLAUDE_PLUGIN_ROOT}/references/dev/severity-and-rubrics.md` — ℹ️ INFO is this command's audit-only observation level. Severity here is **WCAG level × user impact and reach**: is an assistive-tech or keyboard user *blocked* or merely *hindered*, and is the surface a core flow or peripheral? A HIGH+ needs the concrete failure scenario written out ("a screen-reader user cannot submit the signup form"); without it, rate lower. One finding per root cause, occurrences listed inside it; in diff scope don't relitigate pre-existing surfaces the change didn't touch.

   | Severity | Bar | Typical examples |
   |---|---|---|
   | 🔴 CRITICAL | User **blocked** on a core flow, no workaround (WCAG A failure on a core path) | Keyboard trap; modal on a core flow with no Escape/trap/focus return; core-flow form unusable with a screen reader |
   | 🟠 HIGH | Real barrier on a primary path; workaround painful | Click-`div` primary action with no keyboard path (2.1.1); content-bearing images without alt in main flows (1.1.1); icon-only core-flow touchable with no label (rn-props) |
   | 🟡 MEDIUM | Hindered, not blocked — or a peripheral surface | Contrast below AA on secondary text (1.4.3); heading jumps (1.3.1); positive tabindex (2.4.3); missing doc lang (3.1.1); muted autoplay without pause (2.2.2) |
   | 🟢 LOW | Hygiene and noise | Decorative image missing explicit `alt=""`; redundant role; minor label wording |

   Each finding carries `file:line`, the WCAG success criterion, impact, concrete fix, and S/M/L effort.

5. **Reconcile against the findings ledger** per `${CLAUDE_PLUGIN_ROOT}/references/dev/findings-ledger.md` (command id `verify:a11y`; `rule` = catalog id above or `axe.<rule-id>`; normalized evidence = the matched element snippet). Follow its integration recipe: read → scan → reconcile → write back, annotate every finding with its bracketed ledger state (NEW / KNOWN / ACCEPTED / REGRESSED — format in the Output block below), honor `--delta` (detail only NEW + REGRESSED, one-line summary of the rest — the verdict is still computed over all open findings), offer user-driven acceptance after the report, and if no ledger exists run ledger-less and offer to initialize it.

6. **Report and verdict.** Findings exist (or scope is `full`) → save the report to `${user_config.workspace_dir}/reports/a11y/YYYY-MM-DD-a11y.md`; a clean diff scan gets an inline summary only. Verdict math runs over **open** findings (accepted ones excluded — except CRITICAL, which never drops out): any CRITICAL or HIGH ⇒ **FAIL**; any MEDIUM ⇒ **PASS WITH WARNINGS**; else **PASS**. Canonical-tier mapping: FAIL = NEEDS CHANGES, PASS WITH WARNINGS = MERGE READY WITH WARNINGS, PASS = MERGE READY.

Fix all 🔴 and 🟠 before merge; an accepted 🔴 still fails the gate.

## Quality checklist

- [ ] Scope stated (diff vs full) and the UI-surface gate applied — non-UI projects exit "n/a", they don't get noise
- [ ] Inapplicable rules marked n/a by idiom; every applicable rule scanned with honest "no finding" where clean
- [ ] Every reported hit confirmed at its `file:line` — no unverified pattern matches
- [ ] Each finding has severity, `file:line`, WCAG SC, concrete fix, effort; HIGH+ carries a written user-impact scenario
- [ ] Runtime pass ran, or its skip reason is stated; nothing was installed unprompted
- [ ] Ledger reconciled and written back; annotations present; `--delta` honored without weakening the verdict
- [ ] Acceptance recorded only on explicit user instruction
- [ ] Verdict derived from open findings per the severity mapping — never vibes

## Handoff

**Chain:** none — solo command outside the default verify chain (flow-level UX runs there via `/cc:verify:design`).
**Solo:** findings to fix → `/cc:plan:task` per root cause. Flow friction, missing states, copy → `/cc:verify:design`. A manual assistive-tech test script (VoiceOver/TalkBack walkthrough) → `/cc:verify:qa`. To make this audit routine (e.g. weekly `--delta`), automate per `${CLAUDE_PLUGIN_ROOT}/references/dev/scheduling.md`.
**Abort rules:** no user-facing surface → "n/a — no user-facing surface" and stop. Empty diff in diff scope → "nothing to scan" and stop. Runtime tooling unavailable → continue static-only and record the gap explicitly — never imply runtime coverage.
