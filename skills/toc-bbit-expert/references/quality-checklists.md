# Quality Checklists — Per-Tool Validation

Run the relevant checklist before presenting any tool's output as complete.
These are the expert's final gate — an analysis that fails a checklist item is not done.

---

## UDEs
- [ ] Each UDE is observable — evidence could be pointed to
- [ ] Each UDE is specific — two people would describe it identically
- [ ] Each UDE states an effect, not a cause or mechanism
- [ ] No UDE assigns blame or contains an interpretation

## CRT
- [ ] Vertical arrows used throughout (cause-and-effect logic)
- [ ] Every link passes the causality test
- [ ] Every link's hidden assumption is explicitly stated
- [ ] All major UDEs trace to one root cause (not many)
- [ ] Root cause is at the policy or paradigm level
- [ ] Convergence is genuine — not forced
- [ ] Entity and connection clarity checked (CLRs applied)

## Cloud
- [ ] Built right to left (actions → needs → goal)
- [ ] Goal is genuinely shared by both sides
- [ ] Needs are real needs, not disguised wants or positions
- [ ] The conflict between wants is genuine — each action truly impacts the opposing need
- [ ] All five assumptions are stated explicitly on each arrow
- [ ] No common cloud errors (see `clouds-conflict-resolution.md`)
- [ ] Druid mapped if conflict keeps returning

## Druid
- [ ] Both base entities are actual behaviours, not concepts
- [ ] Mutually exclusive — can't do both at once
- [ ] Every cause-and-effect link is valid
- [ ] Both sides are actually happening (not just feared)
- [ ] Each goal violation is serious enough to drive a switch
- [ ] No common druid errors (repetition, premature looping, failed solution, tangent, stopping short)

## Injections
- [ ] Each injection is worded as a STATE, not an action
- [ ] All assumptions raised before reversing — not filtered
- [ ] For core conflicts: 10–30 assumptions listed on the key connection
- [ ] Multiple injections developed (at least 2–3)
- [ ] Most impactful injection selected — not just the easiest
- [ ] Each injection explicitly names the assumption it breaks
- [ ] Selected injection satisfies both needs fully (win-win, not compromise)
- [ ] New UDEs either absent or each addressed by a preventative injection
- [ ] Injection tested: "If this were true right now, would it break the connection?"

## FRT
- [ ] Each link validated: causality + assumption + sufficiency
- [ ] Gaps in the CRT identified and filled with additional injections
- [ ] NBRs actively sought — not skipped; legitimate vs. fear assessed
- [ ] For legitimate NBRs: steps graded +/neutral/-, injection placed after positives
- [ ] Goal is reachable through valid causal steps
- [ ] Win-win state confirmed throughout the FRT
- [ ] "Sufficient, not perfect" principle applied — not chasing zero

## PRT + TT
- [ ] Goal stated as a condition, not a task
- [ ] Horizontal arrows used throughout PRT (prerequisite logic)
- [ ] 5–10 obstacles listed (not too few, not clustered into more than 10)
- [ ] Obstacles stated as current conditions, not missing actions
- [ ] Every IO is a Done Statement (present tense, outcome not action, verifiable, sufficient not over-sufficient)
- [ ] Obstacle kept beside each IO; skipped if obstacle is already cleared
- [ ] Each TT step uses Effect → Action → Logic thinking
- [ ] Every TT Expected Effect is a Done Statement
- [ ] Every action has a Why — no step without a logical reason
- [ ] TT actions are sufficient to produce the expected effect (not underpowered)
- [ ] AE fields are blank until after execution — not pre-filled

## Done Statements
- [ ] Each statement is in present tense
- [ ] The outcome is obvious just from reading it — no context needed
- [ ] Someone outside the team can verify it is done
- [ ] It allows any tactic to reach the outcome (not locked to one method)
- [ ] It is measurable or binary — no vague language like "better," "improved," or "more efficient"
- [ ] One outcome per statement — not two goals merged together
- [ ] It describes the state that must exist, not the last action taken
- [ ] Sufficient to clear the obstacle — not over-sufficient

## NCN
- [ ] Every node is a Done Statement (not an action)
- [ ] Horizontal arrows used (prerequisite logic)
- [ ] Every arrow represents a true prerequisite — not a timing preference
- [ ] Timing preferences shown in positioning (levelling), not in arrows
- [ ] Parallel chains visible — nodes at the same horizontal level can run simultaneously
- [ ] Every node has an assigned owner (person, team, or resource category)
- [ ] Time estimates use elapsed time, not touch time
- [ ] Time estimates use the range method (best / worst → midpoint) — no false precision
- [ ] Gaps between nodes reflect real resource constraints, not optimism

## ABC
- [ ] Behaviour is stated as an action, not an outcome
- [ ] Consequences are listed as they impact *this specific person*, not in general terms
- [ ] Each consequence rated 1–5 for impact on this person
- [ ] Magnitude of change defined before adjusting
- [ ] Consequence changes have a real implementation plan (not just "attitude shift")
- [ ] Replacement behaviour (if needed) analysed separately from the original
- [ ] Ethics check applied: willing to tell the person afterward?

## PMCB
- [ ] Behaviour identified as a group pattern, not one individual
- [ ] Chain traced backwards: behaviour → consequences → measure → policy
- [ ] Broken link identified (usually the measure)
- [ ] Fix targets the broken link — not just adding punishment
- [ ] New measure is free of loopholes and aligns with desired behaviour
- [ ] Connected to any relevant druid loopback if applicable

## Scrutiny (CLRs)
- [ ] Entity clarity checked: each statement clear, unambiguous, standalone
- [ ] Connection clarity checked: cause actually produces effect as written
- [ ] Issues flagged with tokens before editing
- [ ] Iterative: reviewed at least twice (review → edit → review)
- [ ] Level of scrutiny matched to the stakes of the decision
